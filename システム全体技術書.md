# discord_ai_bot システム全体技術書

## 1. 概要

discord_ai_bot は Rust 製の Discord AI Bot。単一の実行ファイル（単一プロセス）内で以下の2つの役割を同時に持つ。

1. **Discord Gateway クライアント**（twilight-gateway）— Discordのメッセージ／スラッシュコマンドを受け取り、AIに問い合わせて応答する
2. **Web API サーバー**（axum）— 外部の別サービス（Webフロントエンド）から同じ会話履歴・AI機能を利用するためのHTTP API

両者は同じ `AppState` 相当のデータ（AIクライアント、埋め込みクライアント、履歴DB、オンメモリのチャンネル設定など）を `Arc` で共有しており、**Discord上の会話とWeb API経由の会話は同じ会話履歴テーブルに記録され、同じRAG検索の対象になる**。

なお、Webフロントエンド（ブラウザで動くUI）は本プロジェクトから分離済みで、別サービスとしてこのAPIをHTTP経由で呼び出す構成になっている。フロントエンドの実装は別リポジトリの管轄。

## 4. 起動シーケンス（main.rs）

1. `.env` を読み込む（`dotenvy`）
2. `register_commands()` でDiscordのスラッシュコマンド（`/model`, `/session`, `/search`）を登録
   - Discord REST API (`GET /oauth2/applications/@me` でアプリID取得 → `PUT /applications/{id}/commands` でグローバル登録)
   - `DISCORD_GUILD_ID` が設定されていれば、追加でギルド限定登録も行う（開発時の即時反映用。グローバル登録と重複表示される点は許容している）
3. `AiClient::new()` でVertex AI用サービスアカウント（`GCP_CREDENTIALS_PATH`）を読み込み、Geminiクライアントを初期化
4. Embedding専用に**別途**同じサービスアカウントファイルを読み込み直して `EmbeddingClient` を初期化（`AiClient` 内部のサービスアカウントは非公開フィールドのため共有できず、二重読み込みになっている）
5. `twilight_gateway::Shard` を作成（Intents: `GUILDS | GUILD_MESSAGES | MESSAGE_CONTENT`）
6. `HistoryStore::new("./data/history.db")` でSQLite初期化（テーブルが無ければ作成）
7. `AppState`（web用）を組み立て、`build_router()` でaxumルーターを構築
8. `tokio::spawn` で `127.0.0.1:3000` にバインドしたWebサーバーを別タスクとして起動（Discordイベントループとは独立して並行動作）
9. メインタスクでDiscordイベントループ（`shard.next_event()`）を回し続ける
   - `Ready` … Bot自身のユーザーIDを保存
   - `GuildCreate` … 参加サーバーのログ出力のみ
   - `MessageCreate` … `tokio::spawn` で `bot::handler::handle_message` を非同期実行（イベントループ自体はブロックしない）
   - `InteractionCreate` … `tokio::spawn` で `bot::interaction::handle_interaction` を非同期実行

## 5. Discordメッセージ処理フロー（bot::handler::handle_message）

1. Bot自身の発言は無視（`msg.author.bot`）
2. Bot宛のメンションが無ければ何もしない
3. メンション部分（`<@...>`）を取り除いたテキストを `cleaned` とする
4. 返信（reply）で送られてきた場合、元メッセージの内容を取得する
   - `msg.referenced_message` にキャッシュがあればそれを使う
   - 無ければ `http.message(...)` でDiscord APIから取得
5. テキストが空かつ画像添付・対応テキストファイル添付も無ければ処理を打ち切る
6. `channel_id = "{DiscordチャンネルID}:{現在のセッション名}"` を組み立てる（セッションはオンメモリの `channel_sessions` マップ、未設定なら `"default"`）
7. タイピングインジケーターを送信
8. 添付ファイルをダウンロードし、画像は base64 化して `inlineData` パーツに、対応拡張子のテキストファイルはデコードして文字列として取り込む（8000文字で切り捨て）
9. ユーザー発言を埋め込み化してDB保存 → RAG検索（直近300件から類似度上位3件、閾値0.3）→ 直近10件の履歴取得 → `contents` 配列を構築（過去の関連発言／直近履歴／今回の発言+返信引用+ファイル内容+画像）
10. チャンネルに設定されたモデル名（`channel_models`、未設定は `gemini-3.1-flash-lite`）に応じて Gemini または OpenAI を呼び出す
11. 応答をDiscordに送信（`send_response`）:
    - コードブロックを含み、かつ2000文字超 → `response.md` として添付ファイル送信（分割による構文崩れを防ぐため）
    - 2000文字以内 → そのまま1メッセージで送信
    - 2000文字超でコードブロックを含まない → 1900文字ごとに改行単位で分割して複数メッセージ送信
12. 応答も埋め込み化してDB保存（`author_id="bot"`, `role="model"`）

## 6. スラッシュコマンド処理フロー（bot::interaction::handle_interaction）

Discordのインタラクションは3秒以内の応答が必須なため、Deferred Response（`type: 5`）パターンを使っている。

1. `POST /interactions/{id}/{token}/callback` に `{ "type": 5 }` を即座に送り「考え中...」状態にする
2. コマンドごとに処理:
   - `/model name:<モデル名>` … `channel_models` マップを更新
   - `/session name:<名前>` / `/session delete:<名前>` … セッション切り替え、または `HistoryStore::delete_session` で削除。`name` も `delete` も無指定なら現在のセッションと一覧を返す
   - `/search query:<内容> ai:<bool> count:<件数>` … `ai:true` なら `AiClient::generate_with_search`（Gemini + google_search、モデル固定 `gemini-3-flash-preview`）、`false`（デフォルト）なら `WebSearchClient::search`（Brave Search）
3. `PATCH /webhooks/{app_id}/{token}/messages/@original` で実際の結果に編集

## 7. AI連携

### 7.1 Vertex AI Gemini（ai::client::AiClient）

- 認証: `gcp_auth::CustomServiceAccount`（サービスアカウントJSON、`GCP_CREDENTIALS_PATH`）から `cloud-platform` スコープのトークンを取得
- エンドポイント: `https://aiplatform.googleapis.com/v1/projects/{project}/locations/global/publishers/google/models/{model}:generateContent`
- `generate_with_contents()`: `tools: [{ "url_context": {} }]` を有効にした状態で `contents` 配列をそのまま送信し、`candidates[0].content.parts[0].text` を取り出す。取得できない場合は固定文字列 `"(応答の解析に失敗しました)"` を返す（＝失敗してもエラーにならず処理は継続する）
- `generate_with_search()`: `tools: [{ "google_search": {} }]` を使い、単発クエリで検索付き応答を生成する（`/search ai:true` 用）
- デバッグ用に生のAPIレスポンスを `println!` で全文出力している（本番運用時はログ量に注意）

### 7.2 Embedding（ai::embedding::EmbeddingClient）

- モデル: `gemini-embedding-001`
- エンドポイント: `us-central1` リージョン固定（Gemini本体とは異なるリージョン）
- `task_type` に `RETRIEVAL_DOCUMENT`（保存時）と `RETRIEVAL_QUERY`（検索時）を使い分けている（Vertex AI Embeddingsの非対称検索最適化機能を活用）
- 失敗時は呼び出し側で `unwrap_or_default()` されるため、空のベクトル（長さ0）としてDBに保存されうる。空ベクトル同士のコサイン類似度計算は0除算になり得るが、`rag::rag::cosine_similarity` はゼロ除算チェックを行っていない（Rustの浮動小数点では `0.0/0.0 = NaN` となり、ソート時の `partial_cmp().unwrap()` がパニックする可能性がある。実運用で埋め込み生成が継続的に失敗する場合は注意が必要）

### 7.3 OpenAI（ai::openai::OpenAiClient）

- エンドポイント: `https://api.openai.com/v1/chat/completions`
- `OPENAI_API_KEY` 未設定でもクライアント自体は起動する（空文字列で初期化される）。実際にGPTモデルを選択した時点でAPI側が401エラーを返す想定
- 生レスポンスを `println!` でデバッグ出力

### 7.4 フォーマット変換（ai::convert）

Gemini独自の `{ role, parts: [{text} | {inlineData}] }` 形式と、OpenAIの `{ role, content }` 形式を相互変換する。

- role変換: Geminiの `"model"` → OpenAIの `"assistant"`、それ以外はすべて `"user"`
- テキストパーツが1つだけの場合はシンプルな文字列contentに、画像混在の場合は `type: "text"` / `type: "image_url"`（`data:{mime};base64,{data}` 形式）の配列contentに変換

## 8. RAG（rag::rag）

`search_similar_with_decay()` の計算式:

```
score = cosine_similarity(候補の埋め込み, クエリの埋め込み) × 0.5 ^ (経過日数 / half_life_days)
```

- `half_life_days = 14.0`（半減期14日。14日前の発言は類似度がそのまま使われた場合の半分の重みになる）
- `min_score = 0.3`（このスコアを下回る候補は除外）
- `top_k = 3`
- 対象候補は「直近300件の埋め込み付きメッセージ」（`get_candidates_for_search` で `created_at DESC LIMIT 300`）に限定されており、全履歴を毎回スキャンするわけではない（会話量が増えても検索コストが際限なく増えない設計）

## 9. データストア（strage::history::HistoryStore）

- SQLite（`rusqlite`, `bundled` フィーチャーでSQLite本体を同梱ビルド）
- テーブル定義:
  ```sql
  CREATE TABLE IF NOT EXISTS messages (
      id INTEGER PRIMARY KEY AUTOINCREMENT,
      channel_id TEXT NOT NULL,   -- 実際は "discordチャンネルID:セッション名"
      author_id TEXT NOT NULL,
      role TEXT NOT NULL,          -- "user" | "model"
      content TEXT NOT NULL,
      embedding BLOB,               -- f32配列をリトルエンディアンでバイト列化したもの
      created_at INTEGER NOT NULL   -- UNIXタイムスタンプ(秒)
  );
  CREATE INDEX IF NOT EXISTS idx_messages_channel ON messages(channel_id, created_at);
  ```
- 接続は `Mutex<Connection>` で保護されたシングルコネクション。**非同期ランタイム(tokio)上で同期的なロック＋SQLite I/Oを行っている**ため、書き込みが集中するとそのリクエストを処理しているワーカースレッドをブロックする点は将来的なスケール時の懸念点（現状は個人利用規模のため問題にならない想定）
- 埋め込みは `f32_to_bytes` / `bytes_to_f32` で手動シリアライズ（4バイトLE ×次元数）。専用のベクトルDBは使っていない

## 10. Web API層の位置付け

Web API（`src/web/mod.rs`）はDiscord側のロジック（`bot::handler`, `ai::*`, `rag::*`, `strage::*`）をそのまま呼び出す薄いレイヤーとして実装されている。Discordメッセージ処理とほぼ同じ手順（埋め込み生成→RAG検索→直近履歴取得→contents構築→モデル振り分け→応答保存）を `chat_handler` 内で独自に再実装しており、**共通関数化はされていない**（`handler.rs` の該当ロジックと `web/mod.rs` の `chat_handler` は別コードとして重複している）。仕様変更時は両方に反映する必要がある点に注意。

エンドポイントの詳細仕様（リクエスト/レスポンス形式、エラー挙動など）は別紙「Web API 技術仕様書」を参照。

## 12. デプロイ構成

- 想定ホスト: Rocky Linux系VPS（さくらのクラウド/AWS等）
- systemdによる常駐化
- Caddyによる自動HTTPS化
- fail2banでの防御
- DuckDNSでのIP更新
- 上記をAnsibleでコード化したもの（`ansible-iac`リポジトリ、別管理）を使ってプロビジョニングする運用

Web API部分（`127.0.0.1:3000`）は外部公開せず、Caddyのリバースプロキシ経由で公開ドメインに紐付ける構成を前提とする。Webフロントエンドを別サービス化したことで、このバックエンドは「Discord Bot + 社内向けAPIサーバー」という役割に整理されている。

## 13. 既知の技術的制約・留意点まとめ

1. **`target_channel_id` がプロセス全体で単一** — Web API経由での「操作対象チャンネル」はグローバルな1状態であり、複数クライアントが同時に別チャンネルを操作する用途には向かない
2. **`channel_models` / `channel_sessions` はオンメモリのみ** — プロセス再起動で失われる（DB永続化はされていない）
3. **エラーが握りつぶされやすい設計** — AI呼び出し・埋め込み生成・検索の失敗はほとんどが空値やエラーメッセージ文字列に変換され、HTTPステータスやDiscord側のログでしか検知できない
4. **`chat_handler`（Web API）と `handle_message`（Discord）のロジックが重複している** — 片方だけ修正すると挙動がズレる
5. **SQLite接続がMutex+同期I/O** — 高負荷時にtokioワーカースレッドをブロックしうる（個人利用規模では実用上問題にならない想定）
6. **埋め込みが空ベクトルになった場合のコサイン類似度計算がNaNになりうる** — ゼロ除算ガードが無い
7. **Embedding用のサービスアカウント読み込みが `AiClient` と重複している** — `AiClient` 内部のフィールドが非公開のため、`main.rs` で同じ認証情報ファイルを再度読み込んでいる
8. **モデル名・チャンネルIDなどのバリデーションがAPI層でほぼ無い** — 不正な値を渡しても200が返り、実際のAI呼び出し時に初めて失敗する
